# 多源输入支持方案：视频 / 音频 / 链接 → 博客

## 一、现状结论

**是的，当前项目只能上传视频文件，无法直接上传音频或链接转化为博客。**

具体限制分布在前后端三层：

| 层级 | 文件 | 限制点 |
| --- | --- | --- |
| 前端 UI | `frontend/src/app/page.tsx` | `<input accept="video/*">` 只允许选择视频；`handleDrop` 中 `file.type.startsWith("video/")` 校验拦截音频；没有任何 URL 输入框 |
| 后端接口 | `backend/app/main.py` | `/api/upload` 只接收 `UploadFile`，保存路径固定 `.mp4` 扩展名（`ext = Path(file.filename).suffix or ".mp4"`）；没有 URL 入参 |
| 流水线 | `backend/app/pipeline/nodes.py` | `extract_audio` 调用 `_ffmpeg_extract`，假设输入是视频（用 `-vn` 去视频流）；删除/重处理接口 `delete_video` / `reprocess_video` 也只查找 `.mp4/.mkv/.avi/.mov/.webm/.flv` 扩展名 |
| 数据库 | `backend/app/models/database.py` | `Video` 表只有 `filename` 字段，没有"来源类型"字段，无法区分视频/音频/链接来源 |

**关键技术事实**：流水线的后续 4 步（转录、章节划分、知识提取、博客生成）**只依赖 WAV 音频文件**，与输入是视频还是音频无关。这意味着只要把"任意输入 → WAV"这一步做适配，后续流程完全不用改。这也是改造能低成本落地的根本原因。

---

## 二、设计原则

1. **统一收敛点**：所有输入最终都规范化为 `storage/audio/{task_id}.wav`（16kHz mono），后续 4 步流水线零改动。
2. **最小侵入**：保留现有 `/api/upload` 视频接口不变（向后兼容），新增独立端点处理音频和链接，避免破坏已有调用。
3. **来源可追溯**：数据库新增 `source_type` 字段，记录输入来源，便于前端展示和后续扩展。
4. **能力可选**：链接下载依赖 `yt-dlp`，音频转录依赖现有 Whisper，两者独立失败不影响视频主流程。

## 三、实现方案

### 方案 A：音频上传支持（最简单，2-3 小时）

#### A1. 后端改造

**新增端点** `POST /api/upload/audio`（`backend/app/main.py`）

```python
@app.post("/api/upload/audio")
async def upload_audio(file: UploadFile = File(...), background_tasks: BackgroundTasks = None):
    task_id = str(uuid.uuid4())
    ext = Path(file.filename).suffix.lower() or ".wav"
    # 音频文件存到 AUDIO_DIR 而非 VIDEOS_DIR，避免混淆
    raw_path = AUDIO_DIR / f"{task_id}_raw{ext}"
    async with aiofiles.open(raw_path, "wb") as f:
        while chunk := await file.read(1024 * 1024 * 10):
            await f.write(chunk)

    async with async_session() as db:
        video = Video(id=task_id, filename=file.filename, status="pending", source_type="audio")
        db.add(video)
        await db.commit()

    background_tasks.add_task(_run_pipeline_task_audio, task_id, str(raw_path))
    return TaskResponse(task_id=task_id, status="pending", message="Audio uploaded")
```

**新增流水线入口** `_run_pipeline_task_audio`（或复用现有入口，通过 `source_type` 分发）

核心差异：跳过 `extract_audio` 步骤，直接执行"音频格式规范化"：

```python
async def _normalize_audio(raw_path, task_id):
    """把任意音频格式转为 16kHz mono WAV"""
    audio_path = AUDIO_DIR / f"{task_id}.wav"
    # 复用现有 _ffmpeg_extract 的 ffmpeg 查找逻辑
    # 命令：ffmpeg -i raw_path -ar 16000 -ac 1 -acodec pcm_s16le audio_path
    duration, _ = await _run_in_thread(_ffmpeg_convert_audio, raw_path, audio_path, AUDIO_SAMPLE_RATE)
    return audio_path, duration
```

**数据库迁移**（`backend/app/models/database.py`）

`Video` 表新增字段：
```python
source_type = Column(String(32), default="video")  # video / audio / url
source_url = Column(String(1024), default="")       # 链接来源时记录原始 URL
```

参考现有 `processing_duration` 列的 `ALTER TABLE` 兼容写法（`init_db` 末尾 try/except）。

**删除/重处理接口扩展**

`delete_video` 和 `reprocess_video` 中的扩展名查找列表需要追加音频格式：`[".mp3", ".wav", ".m4a", ".aac", ".ogg", ".flac", ".wma"]`，或者改为按 `source_type` 决定查找目录（视频在 `VIDEOS_DIR`，音频在 `AUDIO_DIR`）。

#### A2. 前端改造（`frontend/src/app/page.tsx`）

**上传区域改造**：把单一上传区改为 Tab 切换或文件选择器 `accept` 扩展：

```tsx
// 原：accept="video/*"
// 改：accept="video/*,audio/*"
<input type="file" accept="video/*,audio/*" ... />
```

**拖拽校验放宽**：
```tsx
// 原：if (file && file.type.startsWith("video/"))
// 改：if (file && (file.type.startsWith("video/") || file.type.startsWith("audio/")))
```

**根据文件类型选择端点**：
```tsx
const isAudio = file.type.startsWith("audio/");
const endpoint = isAudio ? "/api/upload/audio" : "/api/upload";
xhr.open("POST", `${API_BASE}${endpoint}`);
```

**文案调整**：上传提示从"上传视频"改为"上传视频或音频"，并说明音频会跳过音频提取步骤。

### 方案 B：链接输入支持（中等复杂度，4-6 小时）

#### B1. 依赖引入

链接下载需要 **`yt-dlp`**（支持 YouTube / Bilibili / 抖音 / Twitter 等 1000+ 平台）：

```bash
pip install yt-dlp
```

`yt-dlp` 同时支持下载视频和纯音频，且能自动调用 ffmpeg 转码。建议在 `requirements.txt` 中追加。

#### B2. 后端改造

**新增端点** `POST /api/upload/url`：

```python
from pydantic import BaseModel, HttpUrl

class UrlUploadRequest(BaseModel):
    url: HttpUrl
    audio_only: bool = True  # 默认只下载音频，节省带宽

@app.post("/api/upload/url")
async def upload_url(req: UrlUploadRequest, background_tasks: BackgroundTasks = None):
    task_id = str(uuid.uuid4())
    async with async_session() as db:
        video = Video(
            id=task_id,
            filename=str(req.url),
            status="pending",
            source_type="url",
            source_url=str(req.url),
        )
        db.add(video)
        await db.commit()

    background_tasks.add_task(_run_pipeline_task_url, task_id, str(req.url), req.audio_only)
    return TaskResponse(task_id=task_id, status="pending", message="URL submitted")
```

**新增下载+规范化函数**（`backend/app/pipeline/nodes.py` 或新建 `sources.py`）：

```python
def _ytdlp_download(url, task_id, audio_only=True):
    """用 yt-dlp 下载并转为 WAV，返回 (wav_path, duration, title)"""
    import yt_dlp
    audio_path = AUDIO_DIR / f"{task_id}.wav"
    ydl_opts = {
        "format": "bestaudio/best" if audio_only else "best",
        "outtmpl": str(AUDIO_DIR / f"{task_id}_raw.%(ext)s"),
        "postprocessors": [{
            "key": "FFmpegExtractAudio",
            "preferredcodec": "wav",
            "preferredquality": "192",
        }],
        "postprocessor_args": ["-ar", "16000", "-ac", "1"],
        "noplaylist": True,
        "quiet": True,
    }
    with yt_dlp.YoutubeDL(ydl_opts) as ydl:
        info = ydl.extract_info(url, download=True)
        title = info.get("title", "")
        duration = info.get("duration", 0)
    # yt-dlp 输出文件名后缀可能是 .wav.webm 之类，需要重命名
    # 实际产物路径 = outtmpl 去掉 ext + .wav
    return audio_path, float(duration), title
```

**新增流水线入口** `_run_pipeline_task_url`：跳过 `extract_audio`，直接用下载得到的 WAV 进入 `transcribe` 步骤。如果下载失败（链接无效、平台不支持、需要登录等），通过 SSE 推送 `step_error` 事件。

#### B3. 前端改造

**新增"链接"Tab**，上传区域改为三选一：

```tsx
const [uploadMode, setUploadMode] = useState<"file" | "url">("file");

// Tab 切换 UI
{uploadMode === "url" ? (
  <form onSubmit={handleUrlSubmit}>
    <input
      type="url"
      placeholder="粘贴视频或音频链接（YouTube/Bilibili/抖音...）"
      value={urlInput}
      onChange={(e) => setUrlInput(e.target.value)}
    />
    <button type="submit">开始处理</button>
  </form>
) : (
  /* 原文件上传区域 */
)}
```

**链接提交逻辑**：
```tsx
const handleUrlSubmit = async (e: React.FormEvent) => {
  e.preventDefault();
  if (!urlInput) return;
  setPhase("processing");
  const res = await fetch(`${API_BASE}/api/upload/url`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ url: urlInput, audio_only: true }),
  });
  const data = await res.json();
  setTaskId(data.task_id);
};
```

#### B4. 链接支持的风险与边界

| 风险 | 说明 | 缓解措施 |
| --- | --- | --- |
| 平台反爬 | YouTube/Bilibili 可能要求 cookie 或限流 | `yt-dlp` 支持 `--cookies` 参数，预留配置项 |
| 链接失效/私有 | 下载失败 | 捕获异常，SSE 推送 `step_error`，前端友好提示 |
| 大文件超时 | 长视频下载耗时 | 设置下载超时（建议 10 分钟），前端显示"下载中"进度 |
| 版权合规 | 用户下载受版权保护内容 | README 增加免责声明，仅用于个人学习 |
| 沙箱网络 | 部分平台需要代理 | 配置 `HTTP_PROXY` 环境变量透传给 `yt-dlp` |

### 方案 C：统一抽象（推荐，长期最优）

如果三种输入都要支持，建议重构为**统一输入适配层**，避免三个 `_run_pipeline_task_*` 函数重复：

#### C1. 抽象 SourceAdapter

新建 `backend/app/pipeline/sources.py`：

```python
from abc import ABC, abstractmethod
from pathlib import Path

class SourceAdapter(ABC):
    """把任意输入源转换为标准 WAV 文件"""
    @abstractmethod
    async def to_wav(self, task_id: str) -> tuple[Path, float, str]:
        """返回 (wav_path, duration_seconds, title)"""
        ...

class VideoFileAdapter(SourceAdapter):
    def __init__(self, video_path: str): ...
    async def to_wav(self, task_id):
        # 调用现有 extract_audio 逻辑
        ...

class AudioFileAdapter(SourceAdapter):
    def __init__(self, audio_path: str): ...
    async def to_wav(self, task_id):
        # 调用 _normalize_audio 逻辑
        ...

class UrlAdapter(SourceAdapter):
    def __init__(self, url: str, audio_only: bool = True): ...
    async def to_wav(self, task_id):
        # 调用 _ytdlp_download 逻辑
        ...

def get_adapter(source_type: str, **kwargs) -> SourceAdapter:
    mapping = {
        "video": VideoFileAdapter,
        "audio": AudioFileAdapter,
        "url": UrlAdapter,
    }
    return mapping[source_type](**kwargs)
```

#### C2. 统一流水线入口

`_run_pipeline_task` 改为：

```python
async def _run_pipeline_task(task_id: str, adapter: SourceAdapter):
    async with async_session() as db:
        # ... 查询 video 记录
        state = {"task_id": task_id, "adapter": adapter}

        # 第一步：统一适配为 WAV
        wav_path, duration, title = await adapter.to_wav(task_id)
        state["audio_path"] = str(wav_path)
        state["duration"] = duration

        # 后续 4 步完全复用现有代码
        state = await transcribe(state)
        state = await segment_chapters(state)
        state = await extract_knowledge(state)
        state = await generate_blog(state)
        # ... 持久化逻辑不变
```

#### C3. 统一上传端点（可选）

如果想进一步统一，可以把三个端点合并为一个：

```python
@app.post("/api/upload")
async def unified_upload(
    background_tasks: BackgroundTasks,
    file: UploadFile = File(None),
    url: str = Form(None),
):
    if file:
        source_type = "video" if file.type.startswith("video/") else "audio"
        adapter = get_adapter(source_type, file=file)
    elif url:
        adapter = get_adapter("url", url=url)
    else:
        raise HTTPException(400, "Either file or url is required")

    task_id = str(uuid.uuid4())
    # ... 创建记录
    background_tasks.add_task(_run_pipeline_task, task_id, adapter)
    return TaskResponse(task_id=task_id, status="pending")
```

## 四、推荐落地顺序

| 阶段 | 内容 | 预估工时 | 价值 |
| --- | --- | --- | --- |
| **Phase 1** | 方案 A：音频上传支持 | 2-3 小时 | 立即支持播客、录音、会议音频转博客，覆盖最常见需求 |
| **Phase 2** | 方案 B：链接输入支持 | 4-6 小时 | 支持 YouTube/Bilibili 等平台一键转博客，极大降低输入门槛 |
| **Phase 3** | 方案 C：统一抽象重构 | 2-3 小时 | 消除重复代码，为后续支持更多源（如 RTMP 直播流、Google Drive 链接）打好基础 |

**建议先做 Phase 1**：改动最小、风险最低、价值最直接。Phase 2 视实际需求决定，Phase 3 在前两个稳定后再做重构。
